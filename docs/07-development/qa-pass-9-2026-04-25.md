## Critical fix details

### Bug 11 + 12: Onboarding trigger chips broken (CRITICAL)

**Symptom:** Tapping any chip in `Step 7 — Triggers` (Stress / Boredom /
After meals / Social pressure / Emotions / Late-night) had no effect. The
"Continue" CTA stayed disabled forever. Same chip configuration in
mobile-mcp's `mobile_list_elements_on_screen` returned only the screen
title, eyebrow, subtitle, and Continue button — chips invisible to WDA,
to Voice-Over, and to E2E automation.

**Root cause:** Two layered issues:
1. `<Pressable><GlassCard>...</GlassCard></Pressable>` pattern. iOS
   `expo-blur`'s underlying `UIVisualEffectView` consumes touches at the
   native layer and does NOT honor RN's `pointerEvents="box-none"` prop
   set on the BlurView wrapper. Touches never reach the parent Pressable.
2. `accessibilityRole="checkbox"` on the Pressable was not surfacing in
   the WDA element tree — chips invisible to automation/Voice-Over even
   when tappable.

**Fix:** In `app/(onboarding)/quiz/triggers.tsx`:
- Replaced `Pressable + GlassCard` wrapping with a flat `Pressable + View`
  that mimics the glass tint via solid `rgba(255,255,255,0.55)` for off
  state and `rgba(255,172,160,0.4)` + coral border for on state. No more
  BlurView in the touchable path.
- Switched `accessibilityRole="checkbox"` → `"button"` with
  `accessibilityState={{ selected: isOn }}` and label suffix
  `, selected` for on state. Chips now appear as `Button` in the
  element tree and tap reliably.

Also applied the role-only `checkbox → button` swap to:
- `app/(onboarding)/quiz/motivation.tsx`
- `app/(modals)/craving-log.tsx`

(Defensive — same a11y issue would apply to those multi-select chips
even though their visual styling differs.)

`components/ui/GlassCard.tsx` got a defensive `pointerEvents="box-none"`
on the BlurView host and `pointerEvents="none"` on the absolute-fill
background — won't help with the iOS native consumption above, but is
correct for any pure View parent and harmless otherwise.

### Bug 5: Settings email hardcoded placeholder

**Symptom:** Settings → ACCOUNT → Email row showed `s***@gmail.com`
regardless of who was signed in. Maya signed in as
`maya@sugarquit.test`, but Settings claimed `s***@gmail.com`.

**Root cause:** `SECTIONS` was a module-level `const` array with the
email value baked in as a string literal. No subscription to the store.

**Fix:** Pulled SECTIONS into a `buildSections(emailMasked)` factory
called inside the component, with `email` read from `useUserStore` and
masked through `maskEmail()` (first char + `***` + `@domain`). Sign-out
now shows `Sign in to set` placeholder.
